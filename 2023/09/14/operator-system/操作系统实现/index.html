003 主引导扇区

BIOS

Basic Input Output System

BIOS 在加电自检将主引导扇区读 0x7c00 位置， 并跳转到这里执行。

break 0x7c00 表明在 0x7c00 处打断点， 可简写为b 0x7c00。

int 0x10 系统调用， 显示器相关的功能，相关文献 IMB PS 2 and PC BIOS Interface Technical Reference

实模式

又称 8086模式， 16位， 与实模式相对的是保护模式

  • Real Mode
  • Protected Mode

实模式的寻址方式

有效地址 = 段地址 * 16 + 偏移地址

EA = 0xb800 * 0x10 + 0 = 0xb8000

EA (Effective Address)

实模式时寄存器几乎都是16位， 需要访问 1MB 内存， 需要20位，也就是20根地址线，但一个寄存只可以提供16根地址线

20 - 16 = 4， 剩余的地址线由段地址提供

段地址向左移4位, 段地址 << 4, 段地址向左移4位就提供的剩下的4位

保护模式，寄存器变为32位， 可访问4G内存， 就不需要上面这种方式， 段地址被用于了其他的功能

主引导扇区的结构

  • 代码：446B
  • 硬盘分区表：64B = 4 * 16B
  • 魔数：0xaa55 或 0x55, 0xaa

007 内存检测

操作系统不知道要运行在什么机器上， 尽可能做更多的兼容， 跑在更多的机器上。

我们需要知道哪块内存可以用， 哪块内存不可使用， 内存的开始位置和内存的长度, 以及内存的类型

BIOS 系统调用 0x15， 子功能 0xe820

Address Range Descriptor Struct ARDS 地址范围描述符

20 bit 的数据结构

字节偏移量 属性名称 描述
0 BaseAddrLow 基地址的低 32 位
4 BaseAddrHigh 基地址的高 32 位
8 LengthLow 内存长度的低 32 位，以字节为单位
12 LengthHigh 内存长度的高 32 位，以字节为单位
16 Type 本段内存的类型

我们的操作系统是 32 bit 的， 所以基地址的高 32 位和内存长度的高 32位是 0

Type 字段

Type 值 名称 描述
1 AddressRangeMemory 这段内存可以被操作系统使用
2 AddressRangeReserved 内存使用中或者被系统保留，操作系统不可以用此内存
3 存储ACPI表，可以被操作系统回收。
4 操作系统不可使用这段内存。
5 已经损坏的内存区域，不可使用。
其他 未定义 未定义，将来会用到．目前保留． 但是需要操作系统一样将其视为ARR(AddressRangeReserved)

调用前输入

寄存器或状态位 参数用途
EAX 子功能号： EAX 寄存器用来指定子功能号，此处输入为 0xE820
EBX 内存信息需要按类型分多次返回，由于每次执行一次中断都只返回一种类型内存的ARDS 结构，所以要记录下一个待返回的内存ARDS，在下一次中断调用时通过此值告诉 BIOS 该返回哪个 ARDS，这就是后续值的作用。第一次调用时一定要置为0，EBX 具体值我们不用关注，字取决于具体 BIOS 的实现，每次中断返回后，BIOS 会更新此值
ES: DI ARDS 缓冲区：BIOS 将获取到的内存信息写入此寄存器指向的内存，每次都以 ARDS 格式返回
ECX ARDS 结构的字节大小：用来指示 BIOS 写入的字节数。调用者和 BIOS 都同时支持的大小是 20 字节，将来也许会扩展此结构
EDX 固定为签名标记 0x534d4150，此十六进制数字是字符串 SMAP 的ASCII 码： BIOS 将调用者正在请求的内存信息写入 ES: DI 寄存器所指向的ARDS 缓冲区后，再用此签名校验其中的信息

EBX 输出为 0 是表示结束

返回值

寄存器或状态位 参数用途
CF 位 若CF 位为 0 表示调用未出错，CF 为1，表示调用出错
EAX 字符串 SMAP 的 ASCII 码 0x534d4150
ES:DI ARDS 缓冲区地址，同输入值是一样的，返回时此结构中己经被BIOS 填充了内存信息
ECX BIOS 写入到 ES:DI 所指向的 ARDS 结构中的字节数，BIOS 最小写入 20 字节
EBX 后续值：下一个 ARDS 的位置。每次中断返回后，BIOS 会更新此值， BIOS 通过此值可以找到下一个待返回的 ARDS 结构，咱们不需要改变 EBX 的值，下一次中断调用时还会用到它。在 CF 位为 0 的情况下，若返回后的 EBX 值为 0，表示这是最后一个 ARDS 结构

008 保护模式

系统刚启动时是 8086 模式 只能访问 1M 的内存， 此时称为 实模式，所有的内存，功能对于应用程序与操作系统都是一样的，病毒可以完全控制计算机，这样很不安全

80286 引入了 16位的保护模式

我们使用的是 32位保护模式

保护模式

  • 信息: IT Information Technology 《信息论》
  • 寄存器 / 有一些寄存器只能被操作系统访问
  • 高速缓存 对程序员不透明
  • 内存 / 使用描述符来保护内存
  • 外部设备 / 硬盘 / in/out

全局描述符

  • 内存的起始位置
  • 内存的长度 / 界限 = 长度 - 1
  • 内存属性 (谁可以用)

cpu 根据描述符来判断是不是需要报错

建议搜一下 80386 与 80286 的图片

type segment = 1
| X | C/E | R/W | A |

  • A: Accessed 是否被 CPU 访问过
  • X: 1/代码 0/数据
  • X = 1: 代码段
    • C: 是否是依从代码段 （是依从代码段应用程序可以直接跳到这个区域来执行代码， 如果不是， 是操作系统的代码， 就会报错， 依从代码段不需要修改特权级， 否则必须用某种手段修改特权级，执行代码
    • R: 是否可读
  • X = 0: 数据段
    • E: 0 向上扩展 / 1 向下扩展 (栈段)
    • W: 是否可写

010 ELF 文件格式简介

Executalbe and Liinking Fort / 可执行和链接的格式

  1. 可执行程序 / python / bash / gcc
    • 直接用 gcc -o 生成的格式就是 可执行程序 底层调用了链接程序 ld 是将 可重定位文件链接成 可执行程序
  2. 可重定位文件 / gcc -c 一般以 .o 为后缀 也就是静态库，可以使用 ar 命令 将多个 .o 文件组织成一个 .a 文件, .a 文件就是一个静态库， .a 文件比 .o 文件检索速度快， 内容差不多

012 编译和链接

源文件(.h .cpp .c) - 预处理 -> 源文件(.c) - 编译 -> 汇编文件(.s .asm) - 汇编 -> 目标文件(.o) - 链接 -> 可执行程序(.out)

预处理

gcc -m32 -E main.c -I../include > test.c
32 位程序 预编译 include的目录 写入test.c

编译

/usr/lib/gcc/x86_64-pc-linux-gnu/13.2.1/cc1

gcc -m32 -S test.c > test.s

汇编

as -32 test.c -o test.o

readelf -a test.o

链接

/usr/lib/gcc/x86_64-pc-linux-gnu/13.2.1/collect2

链接就是给这些地址(Addr)一个值, 让他们连起来

ld -m elf_i386 -static test.o -o test.out -e kernel_init

可重定位文件可塞到任何程序里面

gcc 集成

gcc 将上面编译的过程集成到了一起

`gcc -m32 main.c -I../include -o mian.out -e kernel_init -nostartfiles`

程序在启动或退出时会调用系统调用， startfiles 系统会包装一下入口函数， 调用入口函数之前先调用一些系统调用 或初始化一些环境变量， 参数等， 我们不需要。

`gcc --verbose -m32 main.c -I../include -o mian.out -e kernel_init`

–verbose 显示 gcc 编译的细节。

编译和预处理直接有 cc1 来完成
